## 開発環境

### 1. OS + Launcher

OS は `Windows 11 Pro`、Launcher は `Open-Shell` + `Flow Launcher` を組み合わせた二重構成です。

- Windows 11 Pro
  開発体験を考えると、Home ではなく Pro 以上は選択するのが一般的です。
  今回は使用していませんが、Docker、WSL、Hyper-V といった仮想環境関連の機能を制限なく使える点は大きな利点です。

- `Open-Shell`
  Windows 11 で、Windows 7 以前のスタートメニューを再現するツールです。
  主に、`Wz Filer`というファイラーの起動に使用しています。
  旧来の一覧表示とフォルダー構成による整理が便利です。
  また、よく使うアプリをスタートメニューのトップに表示できるので、起動時間を短縮できます。
  `Wz Filer`をメニューのトップに登録しているので `[Win] → 下カーソル → [ENTER]` の 3 操作で起動できます。

- `Flow Launcher`
  `Flow Launcher` というより `シェルプラグイン` がメインです。
  シェルプラグインをカスタマイズして、`[Win]+R`キーで起動するようにし、コマンド履歴を表示させています。
  コマンド履歴は入力キーにあわせて絞り込み検索されるので、`SystemPropertiesAdvanced.exe`(システムのプロパティ)のような複雑なコマンドも、`sys`による絞り込みで簡単に起動できます。
  通常の Windows では、「設定 → システム → バージョン情報 → システムの詳細設定」から［システムのプロパティ］を開く必要があります。
  途中で設定ウィンドウの待ち時間が発生するため、操作感には大きな差があります。
  環境変数の編集は、[システムのプロパティ]から[環境変数]ボタンをクリックする必要があるので、操作時間にかなりの差が出ます。
  入れておく価値があるツールです。

### 2. AIエージェント

- `Claude Code`
  今月に入って Max プランを契約しました。カスタムコマンドで`GitHub Issue`や`PR`を書かせまくっているので、Pro プランでは全然足りません。
  Claude Code の実行には `bash` が必要なのですが、`Git for Windows`に含まれている`bash`が使用できます。
  ただし、後述する BusyBox に含まれる bash を使用するとエラーが発生するため、以下の環境変数を設定しています。

  ```bash
  CLAUDE_CODE_GIT_BASH_PATH=C:\app\scoop\apps\git\current\bin\bash.exe
  ```

- `ChatGPT`
  プログラムや仕様書のレビューや、アイデアの壁打ちに使っています。
  仕様書駆動開発では、Claude で仕様書作成 → ChatGPT で仕様書を仕上げ → Claude でコーディングというサイクルで使用してます。
  この記事自体、 [`tech-article-reviewer`](https://github.com/atsushifx/tech-article-reviewer/) という自作のレビュワープロンプトでレビューしています。
  Claude でプロンプトを作成し、ChatGPT でレビューして改善点を指摘、そのうえで再び Claude でコーディングするサイクルで使用しています。

### 3. エディタ

エディタは、`Visual Studio Code`, `Wz Editor 5`, `Nvim`の 3種類を適宜、使い分けています。

- `Visual Studio Code`
  メインで使っています。主な用途は、プログラミング時のコーディングや設定ファイルの編集、それと記事の執筆です。
  `VSCode`にはプロファイル機能があるので、プログラミングプロファイルと文章用のマークダウンプロファイルを切り替えながら使ってます。
  プログラミングプロファイルでは、インデントやシンタックスハイライト、言語に応じたオートコンプリートやパス補完 (Path IntelliSense) 機能を使ってます。
  マークダウンプロファイルでは、`markdownlint` と `textlint` の 2つのリンター拡張を入れています。
  リンターが指摘する問題点は、画面下部に表示されるのでそれに対応することで、高品質なドキュメントを作成しています。

- `Wz Editor 5`
  骨董品、その 1。現在、`WZ Editor 10`が出ているのですが、使っているのは 2000年に出た Village Center 時代の古いバージョンです。
  日本語や絵文字を含むドキュメントを保存すると文字化けするという、致命的な不具合があります。
  それでもなお、手になじんでいるため、手放せずに使い続けています。
  英数字だけのドキュメントを編集する分には問題ないから、と言い訳して使い続けています。
  `Wz Editor`の利点の 1つは、`Ctrl+B`による選択モードの On/Off と検索の組み合わせです。
  Wz では検索パターンが指定されていると、通常のページアップ／ページダウンが 1 ページアップではなく検索パターンが存在する行へのジャンプに変わります。
  選択モードと組み合わせることで、柔軟かつ高速にブロックしてのコピー・ペーストを実行できます。
  もう 1つがヒストリカルバッファです。Wz は、コピー・ペースト用のバッファが履歴を持っていて、行1 をカット、行2 をカットと 2回カットした場合は、行2 のペースト、その次は行1 のペーストと履歴の順番でペーストします。これが、地味に便利で、簡単に行の入れ替えができます。
  上記の機能が便利なために、現代のモダンなエディタに移行できないのが痛し痒しと言ったところです。

- `Neovim`
  `git commit`や`git rebase`などのターミナル内での編集に使用しています。
  設定は、基本的なところはいじっていません。`_nviminfo`をキャッシュディレクトリの下に移し、いつも使うディレクトリから隠したくらいです。
  プラグインは、`editorconfig`, `wakatime`, `fern`の 3つを入れています。
  `wakatime`は[WakaTime](https://wakatime.com/) という自分が作業に費やした時間をトラッキングするサービス向けのプラグインで、このプラグインで`Neovim`上で作業した時間をトラッキングします。
  `editorconfig` と `fern` は一般的なプラグインのため、詳細な説明は省略します。

### 4. ターミナル, ファイラー

- `Windows Terminal`
  Microsoft 公式ターミナルエミュレータ。タブ機能があり動作も軽快なため、日常的な用途には十分だと感じています。
  カラーテーマも昔はいろいろ試していましたが、現在はビルトインの `Campbell PowerShell` テーマに落ち着きました。
  そのほか、`PowerShell`をプロファイルの一番上にしていて起動しやすくしています。
  後の設定ではデフォルトのままで、特に変更していません。

- `Wz Filer (非公式ビルド)`
  骨董品、その 2。`Wz Editor`のマクロ機能で作成された 2画面のファイラーです。
  2000年制作と古いので、ファイルやディレクトリに絵文字が含まれているとアクセスやリネームができないという問題を抱えています。
  英数字および日本語には対応しているため、日常的な運用では大きな支障はありません。
  特徴としては、

  1. Explorer の右クリックメニュー連携:
     シェル拡張や、拡張されたメニューが使えます。
     例えば、`ターミナルで開く`を選択すると表示されているディレクトリでターミナルが開きます。
     Link Shell Extension を使って、シンボリックリンクの作成もできます。
     ディレクトリの移動が簡単にできるので、リンクの作成がかなり楽になります。

  2. ディレクトリジャンプの有用性:
     `Alt+L`キーで登録しておいたディレクトリにジャンプできます。
     自分は`~/workspaces`を登録しているので、開発中のソフトウェアや記事執筆用のディレクトリにすぐにアクセスできます。
     また、ネットワーク上のディレクトリ、WSL 上のディレクトリも登録できるので OS 間のファイルコピーもできるのも便利です。

  3. ソースウィンドウへのコピー:
     地味に便利な機能が、この"ソースウィンドウにあわせる"機能です。
     `Wz Filer`は 2画面ファイラーなので、2つのファイル一覧ウィンドウに別々のディレクトリを表示できます。
     この機能は、2つの画面のディレクトリを同期します。
     同期した後に、相手側のディレクトリを子ディレクトリに移動すると、「ホームにあった設定ファイルを、`config`下に移動する」といった操作が簡単にできます。

  同様の機能は他のファイラーでも使用できるでしょうが、慣れたキーバインドで操作できる点が大きく、現状では乗り換えに踏み切れていません。

### 5. Shell

日常操作の速度と再現性を最優先にしたシェル環境です。

- `PowerShell`
  日常的な開発作業を高速化するため、PowerShell にはかなりのカスタマイズを施しています。

  1. 履歴と補完
     `PSReadLine` の Option を以下のように設定しています。

     ```powershell
     Set-PSReadLineOption -PredictionSource HistoryAndPlugin
     Set-PSReadLineOption -PredictionViewStyle ListView
     Set-PSReadLineOption -Colors @{ InLinePrediction = [ConsoleColor]::Cyan }
     Set-PSReadLineOption -HistoryNoDuplicates
     ```

     この設定をすると、コマンド履歴が下に一覧表示され、カーソルで選べるようになります。
     しかも、単なるコマンド入力履歴ではなくプラグインによる予測機能も含めています。
     `cd`と入力するだけで、いつも使う作業ディレクトリ (それも複数から選んで) ジャンプできます。

  2. タブ補完
     入力のタブ補完機能は、PowerShell の Module、Carapace、各ツールの`Completion`機能を組み合わせています。
     以下で、それぞれを説明します。

     - Module:
       主なモジュールは、posh-git, CompletionPredictor、WSLTabCompletion です。
       `posh-git`により、Git のサブコマンド、Option、ブランチが補完できます。`git`コマンドは開発でよく使うので、便利に使わせて貰っています。
       `CompletionPredictor`は、補完に予測機能を追加します。例えば、一覧にカレントディレクトリのドットファイル一覧を表示させたりします。

     - Carapace:
       Carapace は PowerShell だけでなく、bash などの各種シェル環境に対応した補完エンジンです。UNIX 型ツールでのタブ補完で役立ちます。

     - completion.d:

       ```powershell
       Get-ChildItem -Path "$basedir/completion.d/*.ps1" | ForEach-Object { . $_.FullName }
       ```

       スタートアップスクリプト中で、上記のようにして`completion.d/`下のスクリプトを実行させています。このディレクトリでは、各種ツールが提供しているタブ補完スクリプトを実行しています。
       これにより、大部分のツールでタブ補完を実現しています。

  3. スクリプトによる機能追加:
     PowerShell は起動時にスタートアップスクリプトを実行します。このなかで以下の機能アップスクリプトを呼び出しています。

     - `Get-Global-History`
       PowerShell 組み込みの履歴機能ではセッション内の履歴しか閲覧できません。`Get-Global-History`は、全セッションの、つまり過去に使用したコマンドも含めた履歴を出力します。

     - `Execute-Command`
       指定したコマンドを実行します。`-Send`,`-Enter`Option を使用することで、ショートカットキーからコマンドを起動できます。

     - `Execute-History`
       グローバル履歴を表示し、選択したコマンドを実行します。
       `Ctrl+p`で実行できます。

  4. キーバインドの変更:
     PowerShell では、`Set-PSReadLineKeyHandler`でキーバインドを変更できます。
     `Wz Editor` を長年使用してきたため、 Wz ライクな動作をするようにキーバインドを変更しています。
     `Ctrl+Shift+Z`で`exit{ENTER}`を入力するようにしており、ショートカットキーで`PowerShell`を終了できます。

- `BusyBox`
  `BusyBox`をパッケージマネージャーの`Scoop` 経由でインストールしています。
  BusyBox は、Windows 上で UNIX 系ツール群を補完するための中核的なツールです。
  実際、これのおかげで Windows 上で bash シェルスクリプトが動いています。
  Windows と Linux とで同じシェルスクリプトを動かすための鍵となるプログラムです。

### 6. パッケージマネージャー

用途別にパッケージマネージャーを使い分けています。

- `WinGet`
  Microsoft 公式パッケージマネージャー。Windows Terminal などの Microsoft 公式アプリはもちろんですが、数多くの OSS 系ツールも提供されています。
  特徴は、Option でインタラクティブなインストールが選べること、インストール先ディレクトリを指定できることです。
  また、`winget import`コマンドで指定したパッケージを一括インストールできるので、環境の再構築時に重宝します。
  ただし、`winget import` 実行時は、インストール先がデフォルトのディレクトリに固定されます。
  インストール先を指定したい場合は、手動でインストールするか、スクリプトを組む必要があります。

- `Scoop`
  Windows 用の軽量パッケージマネージャー。マニフェストを GitHub リポジトリに置くだけで配布できる、OSS との親和性の高さが特徴です。
  本環境では開発用使っているツール、`delta` (Git 用の diff ビューワ)、`jq` (JSON ビューワ), `Lefthook` (Git フックマネージャー) などを管理しています。
  `WinGet` で配布されていなくても `Scoop` なら配付されているツールも多いので、Web で気になったツールは Scoop でインストールしてみることも多いです。
  `Zed` も Windows 版がリリースされたという記事が出る前に、Preview 版が登録されていたという体験もしました。

- `Volta`
  `TypeScript`で開発をしています。ランタイムである `Node`と`pnpm` のバージョン管理に使用しています。
  `Deno`, `Bun`も `Volta` 経由でインストールしていますが、単に最新バージョンのインストールだけですね。

### 7. dotfiles

GitHub 上の [`dotfiles`](https://github.com/atsushifx/dotfiles/) 上で管理、公開しています。
Windows 用と WSL 用の 2種類の dotfiles を 1つのリポジトリでまとめて管理しています。今回は WSL 用の説明はせず、Windows 用だけ説明します。

大前提として、Windows ではすべての設定を`dotfiles`では管理できません。
`Open-Shell`のように設定をバックアップできるツールもありますが、Windows Terminal や Flow Launcher は、設定のバックアップに対応していません。そのため、インストール後に手作業で設定を変更することになります。
Windows 公式の設定ファイル用フォルダは、`~/AppData/Roaming`なので設定ファイルが管理外になるツールもあります。

本環境の`dotfiles`では、リポジトリを`~/.local/dotfiles`下にクローンしています。
設定ファイル用のディレクトリは、`dotfiles` 下から `~/.config` にジャンクションを張っています。これにより、設定ファイルを実体としてはリポジトリ内に置いたまま管理できます。
`XDG Base Directory`仕様に対応しているツールである `Neovim`や`Git`は、これで設定を管理できます。

同様に、`~/Documents/powershell`にジャンクションを張っています。
これにより、リポジトリ内のスクリプトが PowerShell のスタートアップスクリプトとして認識されるので、PowerShell の設定がリポジトリで管理できます。

### 8. その他

このセクションでは、上記までのセクションでは説明できなかった雑多なカスタマイズを紹介します。

- Ctrl キー
  CapsLock キーを Ctrl キーにリマップしています。以下のレジストリ追加で実現できます。キーリマップソフトは使用していません。

  ```reg
  REGEDIT4

  [HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Keyboard Layout]
  "Scancode Map"=hex:00,00,00,00,00,00,00,00,03,00,00,00,1d,00,3a,00,00,00,00,00,00,00,00,00
  ```

- インストールディレクトリ
  `C:\Program Files`下はディレクトリに空白が入っている、Windows を再インストールすると消えてしまう、という理由から極力使わないようにしています。
  その代わり、使用しているのが`c:\bin`と`c:\app`の 2種類のディレクトリです。
  ツールは再インストール前提ですが、ポータビリティを優先しています。多くのツールは、インストーラー経由でなくても、Path が通っていれば動作するので不自由は感じていません。

  `c:\bin`下には、最低限の実行環境用に、`Wz Editor`や`busybox`をいれています。
  バックアップメディアから C ドライブにディレクトリごとコピーしてパスを設定すれば、ファイラー、エディタ、最低限の UNIX コマンドが動くようになっています。

  `c:\app`下には`Git`などの開発ツールや、`FastCopy`などの各種ユーティリティが入っています。
  バージョン管理のためには、あとで`WinGet`などでインストールし直す必要がありますが、バックアップから C ドライブへコピーすれば、とりあえず使えるツールを入れています。

- ATOK
  ATOK パスポートを使用しています。
  ATOK はいろいろとカスタマイズしていますが、ATOK Sync のおかげでカスタマイズした設定に戻せます。
  キーカスタマイズでは、 Wz のキーバインドにあわせて`Ctrl+S`,`Ctrl+D`で文節の前後移動、`Ctrl+A`,`Ctrl+F`で文節長さの短縮、伸張ができるようにしています。
  また、`[変換]`、`[ひらがな]`キーだけで日本語入力の On/Off をできるようにしています。

- パス設定
  インストールディレクトリで説明したとおり、`c:\bin`下と`c:\app`下にツールをインストールしています。
  これにあわせ、システム環境変数の Path に上記ディレクトリを登録しています。これにより、Windows Terminal からコマンドを入力すれば、多くのツールが起動します。
  ツールが単体で動作する場合は、`c:\app`にシンボリックを張って Path を追加しなくても起動できるようにしています。
  ユーザー環境変数 Path には、Scoop 用に`~/app/scoop/shims`、WinGet 用に`${LocalAppData}/Microsoft/Winget/Links`を追加しています。

- 一時作業用ドライブ
  `temp`や`works`などの一時ディレクトリ用にドライブを割り当てています。
  1つめは RAMDISK で、ドライブ W に容量 2GB。PowerShell 起動時に、W:が使用可能な場合は、環境変数 `TEMP` および `TMP` に`w:\temp`を割り当てています。
  2つめは、SUBST を使って`c:\temp`にドライブ Z を割り当てています。
  両方のドライブには、システム用に`temp`ディレクトリ、ユーザーの一時作業用に`tmp`と`works`ディレクトリが作成済みとなっています。

## おわりに

はじめは、他の開発環境の記事と同様に、各ツールについて 2〜3 行程度のコメントで済ませるつもりでした。
しかし、AI によるレビューに対応しているうちに文章が次第に長くなり、結果としてこの分量になりました。

特に PowerShell のカスタマイズ部分の分量が多くなりました。
今回の記事ではタブ補完などの UI/UX 関連のカスタマイズだけでした。
プロンプトの変更や作業トラッキングサービス `wakatime` との連携など、書いていないカスタマイズもまだあります。
